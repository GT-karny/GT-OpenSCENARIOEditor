# OpenDRIVE Element Support Matrix — Open/Edit (2026-07-05)

> Generated by multi-agent element-level audit (6 areas x 6 axes, evidence-cited), aligned with GT_Sim dev_v0.12 ODR 1.6-1.9 capabilities.
> Axes: model (shared types) / parse / serialize / viz (3D) / propertyView (panel) / editUI (create-modify-delete).
> since: 1.6 = baseline, 1.7+ = added 1.7/1.8, 1.9 = new in 1.9.
> Companion docs: odr-1.9-gap-matrix.md (XSD delta), ../proposals/opendrive-1.9-support.md (plan).

## CORE + ROAD (header, road attributes/link/type, planView geometry, elevation/lateral profile, surface/CRG, userData/dataQuality/include at core+road scope)

Parse/serialize/model coverage is strong for the 1.6 core+road backbone: header (geoReference, offset, bounds), road id/name/length/junction/rule, road/link (both pred/succ with all attrs incl. virtual-junction elementS/elementDir), all five planView geometry types, elevation, superelevation, shape, surface/CRG, and lossless userData/dataQuality/include all round-trip through parse-road.ts/parse-header.ts/parse-geometry.ts/parse-common.ts and their build-* counterparts. The strongest gap is on the editUI and propertyView axes: only line and arc geometry can be CREATED (via use-road-creation.ts; spiral/poly3/paramPoly3 are load-only, and poly3/paramPoly3 are explicitly read-only in the property editor); elevation, superelevation, shape, surface/CRG, road/type entries (create), road/link forms, and all header metadata beyond name/date/bounds/geoReference have NO editing UI. The AddGeometryCommand/RemoveGeometryCommand engine commands exist and are tested but are wired to NO UI, so geometry insert/delete is partial. Notable model/parse holes even at 1.6: header @version and @vendor are neither modeled, parsed, nor serialized; road/type @country is dropped (not in OdrRoadTypeEntry, not parsed/serialized); road/type/speed @max is forced to a number so the "no limit"/"undefined" string maxSpeed values are lost. On viz, elevation IS applied to the road/lane mesh (lane-boundary.ts evaluateElevation) but superelevation is NOT applied to the road surface mesh — it only affects signal/entity roll in 3d-viewer position resolvers and the 2D cross-section, so the driving surface never banks. All 1.7-1.9 additions in this area (header license, defaultRegulations; road/lateralProfile/crossSectionSurface) are entirely absent (no model/parse/serialize/viz/UI).

## LANES

Lanes have the strongest data-layer support in the codebase: parser (parse-lane.ts) and serializer (build-lane.ts) achieve full round-trip for essentially every OpenDRIVE 1.6 lane element and attribute (laneOffset, laneSection, left/center/right, lane, link, width, border, roadMark incl. type/explicit/sway children, material, speed, access, height, rule). The typed model (packages/shared/src/types/odr-lane.ts) mirrors this fully for 1.6. Weakness is concentrated on the UI and 1.9 axes. In the 3D viewport, only lane WIDTH drives geometry (border is parsed/serialized but never used for boundaries — lane-boundary.ts:23-31 uses width exclusively), lane TYPE drives surface color, and roadMark type/color drive a single flat line with solid/broken dashing; lane height, level (per-lane superelevation suppression), material, double-line road marks, typeDef/explicit/sway line geometry, and the black/violet-in-viz road-mark colors are not rendered. The property/edit UI is the narrowest axis: only lane type, level, width, and a subset of roadMark fields (type/color/width/weight/laneChange) are editable; speed can be edited only if entries already exist (no add/remove); lane.link is read-only; and border, material, access, lane-height, rule, roadMark typeDef/explicit/sway/material/height, laneSection @s and @singleSide have NO property-view or edit UI at all, and no engine command creates them. All ASAM OpenDRIVE 1.7-1.9 lane additions (lanes @layer, laneSection @length, dual-<lanes>, lane @direction/@advisory/@dynamicLaneType/@dynamicLaneDirection/@roadWorks, link @layer, laneValidity @layer, roadMark color 'black', and lane types shared/walking/slipLane) are entirely absent from the model, parser, serializer, viz, and UI. A latent bug: OdrLanePropertyEditor uses underscore road-mark type tokens (solid_solid etc.) whereas the XSD and serializer use space-separated tokens (solid solid), so selecting a compound type in the UI emits a schema-invalid value.

## OpenDRIVE OBJECTS (road/objects/*)

Objects have a strong DATA-MODEL / PARSE / SERIALIZE spine for OpenDRIVE 1.6 but are effectively invisible and uneditable in the app. Every 1.6 element (object + all attrs, repeat, outline/outlines with cornerRoad/cornerLocal, material, parkingSpace, markings with cornerReferences, borders with cornerReferences, objectReference, tunnel, bridge, validity) is typed in odr-object.ts, parsed by parse-object.ts, and serialized by build-object.ts, all wired into road via parse-road.ts:31-42 and build-road.ts:92-101 for a clean round-trip. The weakest axes are viz (ZERO: no object/tunnel/bridge/outline mesh anywhere in packages/opendrive/src/mesh or geometry; RoadGhostPreview.tsx:80 literally sets objects:[]), propertyView (ZERO: OdrPropertyEditor dispatches only road/lane/signal/junction; no object list panel, no property editor, no overlay), and editUI (add/remove ONLY, and NOT user-reachable: AddObjectCommand/RemoveObjectCommand exist and are exposed as store.addObject/removeObject but are called solely by signal-assembly-operations.ts to place signal poles/arms — no web UI ever calls them, and there is no Update command at all). Notable risk: objects survive load/save faithfully but a user cannot see, select, create, modify, or delete them; the only objects that ever exist are auto-generated signal poles, which are themselves not rendered as objects. All 1.7-1.9 additions (perpToRoad/invalidated/temporary, repeat detach/bT/cT/dT, curveLocal, outline-scoped markings, skeleton, object surface/CRG, validity @layer) are entirely absent across all six axes — they are dropped on parse and never emitted.

## SIGNALS + CONTROLLERS (OpenDRIVE road/signals + root controller)

Signals are the single richest OpenDRIVE feature in this editor: v1.6 road/signals/signal is fully typed (packages/shared/src/types/odr-signal.ts:7), parsed (parse-signal.ts:11), serialized (build-signal.ts:10), rendered in 3D via a data-driven (type,subtype)→descriptor catalog (opendrive-engine signal-catalog.ts:162/197) with placement, move, and arm-pole assembly editing, plus a property panel (OdrSignalPropertyEditor.tsx). The weakest spots even inside v1.6 are: (a) many signal attributes round-trip through parse/serialize but are invisible/uneditable in the property UI — hOffset, pitch, roll, unit, countryRevision have NO property view or editor; (b) validity, dependency, positionRoad, positionInertial are typed/parsed/serialized but have zero UI and zero viz effect (dependency/positionRoad/positionInertial are effectively load-and-store-only); (c) the multi-road signalReference element is parsed/serialized but has NO property view, editUI, or viz. The signal <reference> child is a special case: it is created/read programmatically by the arm-pole assembly system (signal-assembly-operations.ts:172,415) and thus reaches viz and indirect editUI, but there is no direct reference editor. Root <controller>/<control> is fully typed/parsed/serialized and has full Add/Update/Remove engine commands (controller-commands.ts) wired into the store (opendrive-store.ts:382-397), but NO UI anywhere in apps/web reaches them — confirmed no add/update/removeController call site outside the engine — so controllers are load/save-only with dead engine commands. All 1.9 additions in this area (length/invalidated/temporary signal attrs; semantics* and all subtypes; staticBoard/vmsBoard/sign/displayArea; vmsGroup/vmsBoardReference; the abstract board/signal_road split) are entirely absent — no types, no parse, no serialize, no viz, no UI. These 1.9 attrs would also be silently dropped on load→save round-trip since there is no generic passthrough.

## JUNCTIONS

Support is solid for the OpenDRIVE 1.6 "common/default" junction only: junction/connection/laneLink/priority/controller/surface-CRG/junctionGroup are all typed (odr-junction.ts, odr-surface.ts), parse (parse-junction.ts), and serialize (build-junction.ts) with round-trip. Auto-junction creation from road intersections is the strongest feature — a full detect→plan→validate→execute pipeline (use-auto-junction-detection.ts, junction-operations.ts, junction-execution.ts, junction-validator.ts) plus manual endpoint-pick creation (use-junction-tools.ts). A visual lane-link editor (LaneLinkEditor.tsx) allows drag-to-add/click-to-remove lane links, the only true structural connection editing in the UI. The weakest areas are ALL non-default junction variants: virtual junctions are catastrophically broken — the mandatory @mainRoad/@sStart/@sEnd/@orientation attributes are not in the OdrJunction type, not parsed, and not serialized, so a loaded virtual junction is silently corrupted on save (they do NOT survive load-edit-save); the "Virtual Junction" UI button (JunctionListPanel.tsx:66) just creates an empty type='virtual' junction with none of its required data. createVirtualJunction/validateVirtualJunctionRefs exist in the engine but are dead code (not exported from index.ts, not reached by any UI). repairJunctionLinks (junction-validator.ts:416) assumes every connection has a connectingRoad/incomingRoad and rewrites incomingRoad from connecting-road predecessor links — for virtual/direct connections (which have no connectingRoad) this either no-ops or, combined with the missing type awareness, risks corrupting hand-authored virtual connections during auto-detection's Phase 4 repair. All 1.7-1.9 additions in this area (direct & crossing junction types, @linkedRoad, @overlapZone, @fromLayer/@toLayer, crossPath, boundary, elevationGrid, roadSection, new junctionGroup type values) are entirely absent across all six axes. 3D viz of junctions is indirect: JunctionMesh renders a fill polygon computed purely from connecting-road + incoming-road boundaries (junction-surface-builder.ts), so any junction without connectingRoad-bearing connections (virtual/direct/crossing/empty) renders nothing.

## RAILROAD + STATIONS + CROSS-CUTTING MISC (road/railroad, station, root controllers, userData/dataQuality/include, geoReference/offset, document collections, header version axis)

Railroad and station support is a thin parse/serialize/model layer with zero downstream reach: OdrRailroad/Switch/mainTrack/sideTrack/partner and Station/Platform/Segment are fully typed (packages/shared/src/types/odr-railroad.ts), parsed (parse-railroad.ts) and serialized (build-railroad.ts), but nothing renders them in 3D and no property panel or edit UI exists for any of them — the OdrPropertyEditor only branches on junction/signal/road/lane/header/geometry (OdrPropertyEditor.tsx:16,58-80). The prior finding that empty <railroad> is dropped is CONFIRMED (parse-railroad.ts:41 returns undefined when switches is empty, and road-level additionalData like userData inside an otherwise-empty railroad is lost). The prior "segment side mis-typed" finding does NOT reproduce — odr-railroad.ts:46 correctly types side as 'left'|'right' matching e_station_platform_segment_side; the only weakness is a silent default of 'left' for the required attribute (parse-railroad.ts:59). Root-level controllers are the strongest cross-cutting item: fully modeled/parsed/serialized AND backed by Add/Remove/UpdateControllerCommand + store actions, but no UI reaches those actions (grep of apps/web finds only OpenSCENARIO controllers), so editUI is partial. junctionGroups and stations are round-tripped as document collections but have zero UI/viz. The biggest cross-cutting gap is g_additionalData (userData/dataQuality/include): preserved ONLY on header and road scopes; silently dropped on lane, object, signal, junction, station, switch, controller, geoReference, offset, and the OpenDRIVE document root. On the header version axis, revMajor/revMinor flow parse→serialize losslessly (loadDocument replaces the whole doc) but are read-only in the UI (OdrHeaderPropertyEditor.tsx:38-51) and hardcoded to 1.6 for new files (defaults.ts:29); header version and vendor attributes are absent from the model and dropped entirely, and offset is modeled/parsed/serialized but has no UI. The parser is strict-whitelist: any element/attribute not explicitly read is discarded (no passthrough), so all unknown document content is normalized away on save.

## 3. Priority implications — top 10 editor work items by end-to-end (load→edit→simulate) value

The simulator is now ahead on 1.9. The dominant risk theme: **the editor's strict-whitelist parser silently destroys the exact 1.9 data the sim now consumes.** Ranking weights (a) round-trip data-loss severity, (b) whether the sim actually *acts* on the data, (c) blast radius.

1. **Virtual-junction attribute round-trip (`@mainRoad/@sStart/@sEnd/@orientation`).** *Highest — ✅ DONE (2026-07-05 model/parse/serialize; 2026-07-07 P2 editUI).* Sim has full native L1–L5 routing; editor silently corrupts them on any save (confirmed in `odr-junction.ts`). A user who loads a valid virtual-junction map and saves gets a broken file the sim then mis-routes. Model+parse+serialize is a small, contained change — pure round-trip fix, no viz needed. Also disarms the `repairJunctionLinks` corruption risk. *(Note: this item's `repairJunctionLinks` corruption-risk disarming was scoped as a validator-exemption fix in Phase 0-B, tracked separately from the P2 modeling work annotated throughout this doc; not independently re-verified in this pass.)*

2. **Generic additionalData / unknown-element passthrough at the document + all-scopes level.** The whitelist-with-no-passthrough parser is the *systemic* cause of nearly every "dropped on round-trip" row across all 6 areas and every 1.9 gap. A passthrough/preservation layer (round-trip unknown children verbatim) would instantly make the editor round-trip-safe for lane layers, semantics, crossPath, crossSectionSurface, curveLocal, VMS, etc. — even before per-feature UI exists. Biggest leverage-per-line.

3. **Surface `loadOpenDrive === false` as a specific `<include>` error + rebuild/redeploy esmini.js.** Cheap, mandatory, unblocks *everything* downstream. The deployed artifact (verified 5,375,824 bytes) is the stale pre-odr_side vintage; without the 6.3MB rebuild the editor's sim runs on the *old* ODR behavior. Also convert the generic toast to a specific "include not supported" message.

4. **Lane-layer / dual-`<lanes>` / @layer round-trip (1.9).** Sim has full L1–L5 + web API. Editor drops the second `<lanes>` element and all layer attrs. High data-loss severity; sim actively selects the permanent layer, so a layered map authored elsewhere is destroyed by the editor.

5. **Signal `<semantics>` + subtypes round-trip (1.9).** Sim uses semantics for L2 behavior (STOP/GIVE_WAY). Editor drops them. Even parse+serialize-only (no UI) restores sim-relevant meaning through load-edit-save.

6. **crossPath / crossing-junction / `<roadSection>` round-trip.** Sim generates crosswalk entities + VD yielding. Editor empties crossing junctions. Round-trip fix prevents silent corruption of pedestrian-crossing scenarios.

7. **Superelevation applied to the road/lane surface mesh (1.6, already parsed).** Data already round-trips; the gap is purely viz (`lane-boundary.ts` calls only `evaluateElevation`). Sim degrades crossSectionSurface *to* superelevation, so banking is doubly relevant. High-value because it's a viz-only change on already-modeled data and makes the driving surface finally match sim behavior.

8. **Fix roadMark compound-type token bug (`solid_solid` → `solid solid`).** Confirmed at `OdrLanePropertyEditor.tsx:43`. Small, but it emits **schema-invalid values** that could trip the sim's stricter 1.9 validation. Correctness fix, tiny surface.

9. **Object visibility + property panel (any 1.6 object viz).** Objects have the richest orphaned data layer — full round-trip, zero viz/UI (RoadGhostPreview sets objects:[]). Sim synthesizes object geometry (bridge/objectReference). Even minimal 3D placeholders + a property list would end the "invisible data" trap and let users see what the sim renders.

10. **Wire the existing-but-dead engine commands to UI (controllers, AddGeometryCommand/RemoveGeometryCommand, road-level).** These are *already implemented and tested* engine commands with no UI call site (root controllers, geometry insert/delete). Lowest implementation cost of any item — pure wiring — unlocks editing capability the codebase already paid for.

*(Items 1–6 are round-trip-safety fixes the sim's new capabilities make urgent; 7–10 are viz/UI value that was latent before the sim bump.)*
